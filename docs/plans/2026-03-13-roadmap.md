## Phase 3: Публичный API + Telegram-бот

**Цель:** Учителя и ученики получают расписание через Telegram. Внешние системы (unified backend школы) могут интегрироваться через API.

### 3.1 Публичный API расписания (без авторизации)

Новый роутер `/api/public/` — доступен без JWT:

- `GET /api/public/timetable/class/{id}?day=1` — расписание класса на день
- `GET /api/public/timetable/teacher/{id}?day=1` — расписание учителя на день
- `GET /api/public/timetable/room/{id}?day=1` — расписание кабинета на день
- `GET /api/public/classes` — список классов
- `GET /api/public/teachers` — список учителей
- `GET /api/public/rooms` — список кабинетов
- `GET /api/public/rooms/free?day=3&period=5` — свободные кабинеты на конкретный урок
- `GET /api/public/teacher/{id}/now` — где сейчас учитель (текущий урок по времени)

Возвращает данные в простом JSON, без пагинации — объём данных небольшой.

### 3.2 Бронирование кабинетов (с API-ключом)

Новая таблица `room_bookings`:
- `id`, `room_id`, `day` (1-5), `period` (1-10), `date` (конкретная дата, не постоянная), `booked_by` (имя), `purpose`, `created_at`
- Бронь на конкретную дату, не постоянная — истекает после этого дня

Эндпоинты (требуют API-ключ в заголовке `X-API-Key`):
- `POST /api/public/rooms/{id}/book` — забронировать (day, period, date, booked_by)
- `DELETE /api/public/rooms/{id}/book/{booking_id}` — отменить
- `GET /api/public/rooms/free?day=3&period=5&date=2026-03-14` — учитывает и расписание, и брони

API-ключи создаёт admin в веб-интерфейсе (`GET/POST /api/api-keys`).

### 3.3 Telegram-бот (aiogram 3, reference-реализация)

Бот в `backend/bot/` — отдельный процесс, работает через публичный API Ianus.

**Идентификация:**
- При первом `/start` — выбор: "Я учитель" → список учителей, "Я ученик" → список классов
- Бот запоминает привязку Telegram ID → teacher_id/class_id в своей таблице `bot_users`
- Можно перепривязаться командой `/reset`

**Команды/кнопки:**
- **"Моё расписание"** → расписание на сегодня (или завтра, если уроки закончились)
- **"Следующий урок"** → ближайший урок (предмет, кабинет, время)
- **"Где учитель?"** → выбор учителя → текущий урок (кабинет)
- **"Свободные кабинеты"** → на текущий/следующий урок
- **"Забронировать кабинет"** → выбор кабинета из свободных → мягкая бронь на один урок
- **"Расписание на завтра"** → полное расписание следующего учебного дня
- **"Расписание на неделю"** → сводка на всю неделю

**Интеграция с unified backend школы:**
- Unified backend вызывает те же `/api/public/` эндпоинты с API-ключом
- Бот Ianus — reference-реализация, код можно переиспользовать

## Phase 4: Редактирование расписания

**Цель:** Завуч может менять расписание прямо в веб-интерфейсе, без aSc Timetables.

### Функционал

- **Drag & drop** карточки уроков между слотами (день/урок)
- **Валидация конфликтов** в реальном времени:
  - Учитель уже занят в этом слоте
  - Класс уже занят
  - Кабинет уже занят
- **Подсветка** доступных слотов при перетаскивании
- **Создание/удаление** карточек
- **История изменений** — каждое перемещение записывается в аудит
- **Откат** — возможность отменить последние N изменений

### Задачи Phase 4

| # | Задача |
|---|--------|
| 1 | API: PATCH /api/cards/{id} (перемещение: day, period, room_id) |
| 2 | API: POST /api/cards (создание), DELETE /api/cards/{id} |
| 3 | API: GET /api/cards/{id}/conflicts (проверка конфликтов для слота) |
| 4 | API: аудит каждого изменения карточки |
| 5 | Frontend: drag & drop на TimetableGrid (react-dnd или dnd-kit) |
| 6 | Frontend: подсветка конфликтов при перетаскивании |
| 7 | Frontend: панель создания/удаления карточек |
| 8 | Frontend: история изменений и откат |

## Phase 6: Алгоритм составления расписания

**Цель:** Автоматическое составление расписания с учётом ограничений.

### Подход

Это NP-полная задача. Варианты:
- **FET** (Free Timetabling Software) — open source на C++, можно вызывать как CLI
- **Собственный solver** на основе constraint satisfaction (OR-Tools от Google)
- **Генетический алгоритм** — своя реализация на Python

### Ограничения для solver

- Учитель не может вести два урока одновременно
- Класс не может быть на двух уроках одновременно
- Кабинет не может быть занят дважды
- Учитель имеет дни/уроки когда недоступен
- Максимум N уроков в день для класса
- Окна (пустые уроки посреди дня) минимизировать

### Задачи Phase 6

| # | Задача |
|---|--------|
| 1 | Исследование: FET CLI vs OR-Tools vs свой алгоритм |
| 2 | Модель ограничений (teacher availability, room capacity) |
| 3 | Интеграция solver как backend-сервис |
| 4 | API: POST /api/schedule/generate (запуск генерации) |
| 5 | API: GET /api/schedule/status (прогресс) |
| 6 | UI: запуск генерации, отображение прогресса и результата |
| 7 | UI: сравнение текущего и сгенерированного расписания |
